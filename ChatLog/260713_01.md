# ChatLog — 26/07/13（第1回）

## 作業概要
- 全社員（50名超）向け経費精算アプリを、既存receipt-camera（個人用）とは別の新プロジェクト**receipt-camera-forEmployees**として立ち上げることを決定し、要件定義・アーキテクチャ設計・フェーズ計画を策定
- Phase 0（基盤構築）のうち、Claude側で実施可能な作業をすべて完了（リポジトリ・GAS・DBスキーマ・共通モジュール・手順書）

## 重要な変更点（プロジェクト立ち上げの決定事項）
- **データベースはCloud SQL (MySQL)を採用**。スプレッドシートをDBとして使わない（50人同時アクセス・承認ワークフローのステータス更新に不向きなため）。GASからは標準JDBCサービスで接続。将来Cloud Run移行時もDBはそのまま使える
- **freeeAPIv2ライブラリは本プロジェクトでは不使用**（管理者個人用の実験ライブラリのため。変更も引き続き禁止）。v2の汎用化後継として**freeeAPIv3**を新規作成する（安定後にライブラリ切り出し前提）。freee OAuthエンドポイント接続はv3導入に必要な範囲で承認済み
- **freee経費申請は社員本人名義**（必須要件）→社員ごとのfreee OAuth（Phase B）
- **社内承認ワークフロー**を実装（freeeの承認は1段階のみで不足）。承認段数・経路はマスタで可変。承認完了後にfreee登録
- 社員マスタ（employees）・ログ（app_logs）は**全社共通マスタ基盤**として他アプリと共用できる設計
- 認証はGoogleログイン（GIS IDトークン→GAS検証→社員マスタ照合）。Phase Bまでfreee接続コードは一切書かない
- 作成済み: PWA雛形（React+Vite）／GAS雛形（`db.js`=JDBCアクセス・`logger.js`=汎用ログ・`main.js`=ping疎通のみ）／DBスキーマ初版（6テーブル: employees, receipts, approval_routes, approval_steps, freee_uploads, app_logs）／`.agents/AGENTS.md`／`docs/SETUP.md`

## 提案事項
- Supabase vs Cloud SQLの比較を提示 → ユーザーが**Cloud SQL (MySQL)**を選択（GCP統一・約$10/月・Cloud Run相性）
- GASの制約（1実行6分・同時実行約30）とCloud Runの説明を実施 → 当面GASで開始し、DBを先に外部化しておくことで将来の移行を容易にする方針で合意

## 技術的メモ
- GitHub Pagesは`gh api`で`build_type=workflow`を設定して有効化。初回デプロイ成功（HTTP 200）
- `clasp create`はclasp 3.xでは`--type webapp`が使えず「Invalid container file type」になる → `--type standalone`で作成（Webアプリ動作はappsscript.jsonの`webapp`設定で制御されるため問題なし）
- GAS Webアプリの匿名公開（`ANYONE_ANONYMOUS`＋`USER_DEPLOYING`）は、**オーナーがGASエディタで一度スクリプトを承認するまで「アクセス権が必要です」を返す**（claspデプロイでは承認ステップが飛ぶため）。`testDbConnection`の初回実行時の承認で解消される見込み
